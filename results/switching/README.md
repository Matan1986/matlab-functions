# results/switching/ README

## Purpose
`results/switching/` is the run-lineage container for Switching execution outputs. It is the primary location for run-scoped evidence and intermediate products.

## Run Container Contract
Expected run pattern:
- `results/switching/runs/<run_id>/`

Each run container should preserve lineage artifacts, including:
- run manifest (for example `run_manifest.json` or equivalent)
- execution status (for example `execution_status.csv` or equivalent)
- run logs
- config/entrypoint snapshot or immutable reference
- run-scoped tables, reports, figures, and diagnostics generated by that run

## What Belongs in Run-Scoped Outputs
- raw outputs and intermediate artifacts
- failed/partial outputs with explicit status labeling
- diagnostic artifacts tied to the run
- per-run family outputs before durable promotion

## Family Indexing Expectations
Within run lineage, outputs must be indexed/tagged by family without merging:
- `legacy_old`
- `canonical_residual_decomposition`
- `canonical_geometric_decomposition`
- `canonical_replay`

Geocanon outputs stay in the geometric decomposition scope and must follow geocanon naming constraints from `docs/switching_artifact_policy.md`.

## Promotion and Safety Rules
- Durable promotion targets are `tables/switching/`, `reports/switching/`, and `figures/switching/`.
- Promotion requires lineage backlinks to source run and producer script.
- Do not delete, move, or rename run folders without explicit lineage and downstream-consumer checks.
- Run folders are lineage evidence, not cleanup staging.
