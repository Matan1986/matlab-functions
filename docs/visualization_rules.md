# Visualization Rules

Use these standards for all figures generated by analysis, diagnostics, validation, and agent-authored scripts in this repository.

## Visualization principles

- Prefer figures that explain the physics clearly over figures that maximize density.
- Match the plot type to the question being answered.
- Use direct labeling, legends, and annotations to reduce ambiguity.
- Prefer one figure per file.
- Avoid large multi-subplot grids unless necessary so figures remain easy to inspect and edit individually.
- Avoid decorative choices that make interpretation harder.

## Multi-curve plotting rules

- If `number_of_curves <= 6`, use distinct line colors or styles with an explicit legend.
- If `number_of_curves > 6`, use a colormap with a labeled colorbar.
- Keep line ordering meaningful, for example by temperature, current, field, or wait time.
- Use thicker lines for primary curves and lighter supporting lines only when necessary.

## Legend vs colormap rules

- If `number_of_curves <= 6`, use an explicit legend.
- Do not use colormaps when `number_of_curves <= 6`.
- If `number_of_curves > 6`, use a colormap plus a labeled colorbar.
- When a colorbar is used, its label must state the physical parameter and units.

## Colormap selection

- Default colormap: `parula`
- Avoid rainbow maps such as `jet`.
- Use diverging maps only when the data are centered on a meaningful zero or signed reference point.
- Keep color meaning consistent across related figures in the same analysis.

## Heatmap conventions

- Use `axis xy` so axes read in the natural direction.
- Always include a labeled colorbar.
- Label both axes with quantity and units.
- State in the title or caption what the color scale represents.
- Use smoothing or interpolation cautiously and disclose it when applied.

## Derivative handling

- Apply light smoothing before computing derivatives.
- Use the minimum smoothing needed to suppress obvious numerical noise.
- State the smoothing method or window in the figure caption, notes, or agent report.
- Do not present derivative maps dominated by noise if the underlying signal is not interpretable.

## Readability standards

- `LineWidth >= 2`
- `FontSize >= 14`
- Axis labels must include units.
- Titles should describe the physical content, not just the script name.
- Tick labels, legends, and colorbars must remain readable at saved resolution.

## Avoiding spaghetti plots

- Do not overlay too many curves in one panel when the result is unreadable.
- Split data into panels, slices, representative subsets, or summary statistics when needed.
- Highlight the most important curves and de-emphasize secondary ones.
- Prefer small multiples, heatmaps, or summary envelopes over dense line stacks when many traces are present.

## Figure export standards

- Save every figure as both PNG and FIG.
- PNG exports should use publication-quality resolution.
- FIG exports must be kept so figures remain editable.
- Recommended export pattern:

```matlab
exportgraphics(gcf,'figure.png','Resolution',300)
savefig(gcf,'figure.fig')
```

## Figure Window Naming (STRICT)

This is a mandatory repository-wide rule for all new figure-producing code.

Required behavior:

- Do not use `title(...)` to encode file identity or figure identity.
- Every figure must set a window name via the figure `Name` property.
- Every figure must set `'NumberTitle','off'`.
- The figure `Name` and the saved filename base (without extension) must be programmatically reused from the same variable.
- Defining separate string literals or separate variables for figure `Name` and saved filename is forbidden, even if the text appears identical.

Recommended pattern:

```matlab
base_name = 'switching_alignment_heatmap';
fig = figure('Name', base_name, 'NumberTitle', 'off');
imagesc(data);
xlabel('Temperature (K)');
ylabel('log_{10}(t / s)');
save_run_figure(fig, base_name, run_output_dir);
```

Correct example:

```matlab
base_name = 'switching_alignment_heatmap';
fig = figure('Name', base_name, 'NumberTitle', 'off');
% ... plotting commands ...
save_run_figure(fig, base_name, run_output_dir);
```

Forbidden behavior:

```matlab
% Forbidden: title used as naming identity
fig = figure;
title('switching_alignment_heatmap');
save_run_figure(fig, 'switching_alignment_heatmap', run_output_dir);

% Forbidden: separate string definitions (independent sources of truth)
fig = figure('Name', 'switching_alignment_heatmap', 'NumberTitle', 'off');
save_run_figure(fig, 'switching_alignment_heatmap', run_output_dir);

% Forbidden: almost-identical names
base_name = 'switching_alignment_heatmap';
fig = figure('Name', base_name, 'NumberTitle', 'off');
save_run_figure(fig, 'switching_alignment_heatmap_v2', run_output_dir);

% Forbidden: suffix drift (_final, _v2, etc.)
base_name = 'switching_alignment_heatmap';
fig = figure('Name', base_name, 'NumberTitle', 'off');
save_run_figure(fig, [base_name '_final'], run_output_dir);

% Forbidden: NumberTitle left enabled
base_name = 'switching_alignment_heatmap';
fig = figure('Name', base_name);
save_run_figure(fig, base_name, run_output_dir);
```

Rationale:

- Single-variable reuse creates one source of truth and prevents accidental drift between figure-window identity and exported artifact identity.
- Forbidding independent definitions eliminates near-match mistakes (`_v2`, `_final`, typo variants) that break traceability during review and post-run inspection.
- Keeping naming metadata in figure properties avoids overloading scientific plot titles with artifact identity.
## Agent visualization self-check

Agents generating figures must include a `Visualization choices` section in their report stating:

- number of curves
- legend vs colormap
- colormap used
- smoothing applied
- justification

Before finalizing a figure set, agents should verify:

- the plot choice matches the number of curves
- labels include units
- fonts and lines are readable
- derivatives are not dominated by noise
- both PNG and FIG exports were produced
